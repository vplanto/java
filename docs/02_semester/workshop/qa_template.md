## Ціль ролі QA

Ви — **інженер якості**. Ваше завдання — **до появи коду** перетворити вимоги на **перевірювані сценарії**: зрозуміло, *що* маємо довести тестом, *на якому рівні* піраміди (Unit / Integration / E2E) і *як це пов’язано з AC та доменними правилами*.

Кожен тест-кейс — це **окремий тікет** `TC-*` у [GitHub Project](https://github.com/users/vplanto/projects/1). Тікет = ваш артефакт перевірки.

> **Зв’язок з Лекцією 3.** Acceptance Criteria — договір PO ↔ Dev; тест-кейс — конкретизація AC у кроки з очікуваним результатом.
>
> **Зв’язок з Лекцією 6.** STLC починається з аналізу вимог; більшість перевірок — **Unit + Integration**, критичні user journeys — **E2E / ручний smoke**.
>
> **Зв’язок з Лекцією 8.** Використовуйте техінки **EP (класи еквівалентності)** та **BVA (граничні значення)** для пошуку сценаріїв (особливо для полів JSON та лімітів). Формулюйте тікет за патерном **AAA (Arrange-Act-Assert)**: чіткі передумови, одна дія та перевірка результату в БД чи API.
>
> 📝 **Naming Convention:** [Конвенція оформлення тікетів](plan.md#-конвенція-оформлення-тікетів-github-project). Заголовок англійською, тіло українською.
>
> 🤖 **Vibe Coding Tip (QA):** Використовуйте AI для генерації **Boundary Value Analysis** сценаріїв.
> **Промпт:** *"Я маю опис поля: [напр. вік від 18 до 65]. Згенеруй список граничних значень для тестування (включаючи негативні та edge-cases) у форматі Markdown таблиці."*
> Це допоможе вам не пропустити жодного "підступного" значення при проектуванні тест-кейсів.

## Робочий процес

### Крок 1: Вхідні артефакти (аналог STLC — Requirement Analysis)

Перед написанням TC перегляньте:
- **User Stories** → [Product Backlog](product_backlog.md): формулювання «Як … хочу … щоб …».
- **Acceptance Criteria** → у тілі Story або в окремих нотатках PO: умови **Given / When / Then** (розділ 5 у [Лекції 3](../03_requirements.md)).
- **SME** → `DR-*`, `EC-*`, `BAU-*`: правила та крайні випадки, які **обов’язково** мають мати негативні або граничні тести.
- **UX** → `UF-*`, `WF-*`: кроки flow для **E2E / ручних** сценаріїв.
- **Dev** → `TT-*`: які шари реалізуються (API, Domain …) — підказує, де логічні **Unit** vs **Integration**.

### Крок 2: План рівня спринту (коротко, усередині першого TC або wiki-нотатки)

На спринт зафіксуйте 3–5 рядків:
- **Scope:** які US закриваємо.
- **Розподіл за пірамідою:** скільки орієнтовно Unit / Integration / E2E / Manual (розділ 2 у [Лекції 6](../06_qa_strategy.md)).
- **Регресія / smoke:** що повторно проганяємо після змін (розділ 5 у [Лекції 6](../06_qa_strategy.md)).

### Крок 3: Створіть тікети `TC-*`

На кожну **перевірювану** гіпотезу (щасний шлях, порушення DR, граничні значення, NFR з backlog) — **окремий** TC.

## Тип тікету

### 🏷 `TC` — Test Case

Один сценарій = один TC. Формулювання має бути **атомарним і перевірним** (ознаки хорошої вимоги — розділ 2 у [Лекції 3](../03_requirements.md)).

**Формат заголовку:** `TC-## EP-XX US-YY | Short English description`

**Тіло тікету (українською):**

```
**Тип:** Test Case
**Сквад:** Alpha / Beta / Gamma
**Пов’язані Stories:** US-XX
**Labels:** test-case, squad:xxx, EP-XX

**Трасованість (traceability):**
- User Story: US-XX — <коротко суть>
- AC / вимога: <номер або цитата Given/When/Then>
- Домен / edge: #DR-.., #EC-.., #BAU-.. (якщо є)
- UX flow: #UF-.. (якщо E2E / ручний)
- Реалізація (опційно): #TT-..

**Рівень тесту (піраміда):** Unit / Integration / E2E / Manual
**Підхід (ящик):** White Box / Grey Box / Black Box — див. розділ 2 у [Лекції 6](../06_qa_strategy.md)

**Пріоритет:** Critical / High / Medium / Low

**Передумови (Given):**
- <стан системи, дані, ролі>

**Кроки (When):**
1. …
2. …

**Очікуваний результат (Then):**
- <вимірюваний результат: код відповіді, поле JSON, повідомлення, стан БД>
- **PASS:** …
- **FAIL:** …

**Негатив / границі (якщо застосовно):**
- <наприклад, порушення DR-04, null, max+1 — узгоджується з EC та [техніками кейсів](../08_test_cases.md)>

**Автоматизація:** Так / Ні
**Інструмент / шар CI (якщо Так):** JUnit, MockMvc, Testcontainers, … — узгодити з [пайплайном](../06_qa_strategy.md) (розділ 5)

**Залежності:** #<пов’язані тікети>
```

## Чеклист якості TC (перед створенням тікета)

Відповідає критеріям хорошої вимоги з [Лекції 3](../03_requirements.md) (розділ 2):

- [ ] **Атомарність:** один TC — одна перевірювана гіпотеза
- [ ] **Перевірність:** є чіткі PASS/FAIL, без «все виглядає ок»
- [ ] **Зрозумілість:** Dev і QA прочитають однаково Then
- [ ] **Трасованість:** вказано US і джерело (AC / DR / EC / UF)
- [ ] **Рівень:** свідомий вибір місця в піраміді, а не «завжди E2E»

---

## Чеклист перед завершенням спринту (роль QA)

- [ ] Для обраних US є TC на **щасний шлях** і на **ключові негативи** (DR / EC)
- [ ] Більшість сценаріїв покриваються **Unit / Integration** ([піраміда](../06_qa_strategy.md), розділ 2)
- [ ] Критичний journey має **smoke** TC (ручний або автоматизований)
- [ ] Створено ≥ 5 тікетів `test-case` у GitHub Project
- [ ] Для автоматизованих TC зазначено місце у **CI** (Fail Fast, [Лекція 6](../06_qa_strategy.md), розділ 5)
