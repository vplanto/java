# Фасилітація: Daily Stand-up (Щоденна синхронізація)

**Роль:** Scrum Master / Викладач (фасилітатор)
**Тривалість:** Строго 15 хвилин
**Ціль церемонії:** Забезпечити прозорість прогресу всередині скваду, виявити міжкомандні залежності та блокери до того, як вони зупинять спринт.

> 📋 **Теоретична основа (з лекційного матеріалу):** Daily Scrum — це не статус-репорт для Scrum Master або викладача. Це **"Distributed System Health Check"** — перевірка здоров'я розподіленої команди, де кожен розробник є вузлом, який репортує свій стан. Якщо вузол "мовчить" (не повідомляє про проблему) — система думає, що він здоровий, хоча насправді він вже впав.

---

## Завдання для студентів (постановка)

Перед Daily кожен учасник скваду готує відповідь на **3 ключові питання**:

1. **Що я зробив від моменту останньої зустрічі?**
   Тікети, переведені в статус «Done»; розроблені артефакти (SME/UX/QA/OPS-шаблони); написаний або відревʼюваний код.

2. **Що я планую зробити до наступної зустрічі?**
   Задачі, які беруться в роботу сьогодні; фокус на вертикальному зрізі User Story (а не горизонтальному шарі).

3. **Які є блокери, що гальмують або унеможливлюють роботу?**
   Технічні перешкоди; очікування на рішення або артефакт від іншої ролі/скваду; неясність вимог; відсутність доступу.

### Обов'язковий формат звіту

```
Команда (Сквад): [Alpha / Beta / Gamma]
Роль у спринті: [SME / UX / DEV / QA / OPS]

Вчора:   ... (1–2 речення)
Сьогодні: ... (1–2 речення)
Блокери:  [Опис або "Відсутні"]
```

> ⚠️ **Важливо:** Якщо ваш сквад заблокований іншою командою (наприклад, Beta чекає на API-контракт від Alpha) — **обов'язково** вкажіть це в блокерах. Замовчування блокера підставляє весь сквад.

## Сценарій проведення (для фасилітатора)

### Крок 1. Відкриття — задати правильний tone (2 хв)

> *«Перед тим як починати, нагадаю головне правило. Daily — це не звіт для мене. Це синхронізація для вас. Якщо ви не повідомляєте про блокери, решта скваду думає, що робота йде за планом — і планує свої задачі, спираючись на ваш "уявний" прогрес. Daily — це страховка від таких помилок.»*

**Перевірте наявність звітів.** Якщо частина учасників не залишила письмовий звіт — **не скасовуйте зустріч і нікого не сваріть**. Перейдіть до Кроку 2.

---

### Крок 2. Робота з відсутніми звітами

Тих, хто не написав звіт, **викликайте відповідати голосом прямо на зустрічі**:

> *«Ви не залишили звіт. Уявіть, що ми на реальному зідзвоні в компанії. Дайте відповідь на 3 питання зараз: що робили, що плануєте, які блокери?»*

**Навчальна мета цього прийому:** показати студентам, що письмовий звіт — це лише інструмент підготовки до *живої* синхронізації, а не самоціль. Daily повинен відбутися за будь-яких умов.

---

### Крок 3. Обхід сквадів (10 хв)

Заслуховуйте звіти по колу у кожному скваді. **Слухайте звіти через призму залежностей між ролями та командами.**

#### Внутрішньокомандні залежності (всередині скваду)

Якщо DEV повідомляє, що сьогодні завершує API-ендпоінт, одразу запитайте QA:

> *«QA, ви готові завтра почати тестувати цей ендпоінт? У вас є готові тест-кейси від SME?»*

Це навчає команду працювати **паралельно**, а не «каскадом» (спочатку DEV → потім QA → потім OPS).

#### Міжкомандні залежності (між сквадами Alpha / Beta / Gamma)

Уважно відловлюйте блокери між скводами. Якщо Сквад Beta залежить від ідентифікатора користувача, який розробляє Сквад Alpha:

> *«Beta, ви отримали контракт від Alpha? Ні? Тоді це ваш головний блокер на сьогодні. Alpha, коли можете надати контракт — сьогодні чи завтра? Зафіксуємо це.»*

---

### Крок 4. Відловлювання антипатернів у реальному часі

Зупиняйте учасників, якщо чуєте класичні антипатерни Daily:

| Антипатерн | Що кажете |
| :--- | :--- |
| **«Звітність»**: *«Вчора я написав 400 рядків коду...»* | *«Кількість рядків не важлива. Яку конкретно User Story або Task ти просунув ближче до статусу Done?»* |
| **«Самодіяльність»**: *«Сьогодні буду оптимізувати базу даних...»* | *«Чи є ця задача в Sprint Backlog? Якщо ні — чому ти береш її замість пріоритетних задач?»* |
| **«Хибне благополуччя»**: *«Блокерів немає»*, але прогресу теж немає | *«Якщо блокерів немає, чому задача досі не передана на рев'ю або в QA?»* |
| **«Технічна дискусія»**: DEV і OPS починають сперечатись про деталі | *«Це технічне обговорення. Daily — не місце для цього. Беремо в **Parking Lot**. Після Daily DEV і OPS залишаються і вирішують це окремо.»* |

---

### Крок 5. Закриття та призначення Parking Lot (3 хв)

> *«Дякую. Зафіксуємо підсумки. [Перераховуєте виявлені блокери]. Ось Parking Lot на сьогодні — питання, яке DEV та OPS зі скваду Beta вирішать відразу після цього мітингу. Хто ще хоче щось додати — 30 секунд. Наступний Daily завтра о тій самій годині.»*

**Контрольні точки після Daily:**

- [ ] Чи всі блокери виявлені та зафіксовані?
- [ ] Чи призначені відповідальні за вирішення кожного блокера?
- [ ] Чи розуміють учасники різних ролей (UX ↔ DEV ↔ QA), хто від кого залежить на сьогодні?
- [ ] Чи немає «мовчазних вузлів» — учасників, які нічого не сказали про прогрес?

---

## Теоретичний контекст (для розбору після зустрічі)

### Чому Daily — це не «домашнє завдання»

Одна з типових помилок починаючих команд — сприймати письмовий звіт як самоціль. Насправді це **артефакт підготовки**, а не сам Daily. Справжня цінність церемонії — жива синхронізація, де виявляються залежності, які неможливо побачити в асинхронному чаті.

### Роль Scrum Master — «Servant Leader»

Scrum Master під час Daily — це не контролер і не менеджер. Він **слуга-лідер** (Servant Leader):

- **Не** оцінює якість роботи учасників.
- **Не** вирішує технічні проблеми замість команди.
- **Допомагає** команді побачити власні «вузькі місця» в комунікації.
- **Навчає** ролі домовлятися між собою (UX ↔ DEV ↔ QA ↔ OPS).

### Матриця залежностей сквадів (довідка)

| Якщо сквад | Залежить від | Артефакт залежності |
| :--- | :--- | :--- |
| Beta (Logic) | Alpha (UX) | API-контракт / JSON-схема ідентифікатора |
| Gamma (Ops) | Beta (Logic) | Docker-образ або специфікація розгортання |
| Alpha (UX) | Beta (Logic) | Фінальна структура відповіді API для екранів |
| Будь-який QA | DEV того самого скваду | Готовність ендпоінту до тестування |

> 📌 **Нагадування:** Scrum Master не відповідає за технічний зміст рішень. Якщо виникає технічна суперечка — це в **Parking Lot**, а не в Daily.
