# Фасилітація: Sprint Review (Демо Інкременту)

**Роль:** Scrum Master / Викладач (фасилітатор), Product Owner (приймає інкремент)
**Тривалість:** 45-60 хвилин
**Ціль церемонії:** Продемонструвати стейкхолдерам (Product Owner, викладач, інші сквади) **робочий інкремент** продукту VARTA, перевірити відповідність Definition of Done та зібрати зворотний зв'язок для коригування Product Backlog.

> 📋 **Теоретична основа (з лекційного матеріалу 02_delivery_methodology):** Sprint Review — це огляд **продукту**, а не процесу (процес обговорюється на Retrospective). Мета — показати **живий, робочий продукт** реальним стейкхолдерам. Не PowerPoint зі скриншотами. Не «ми майже готові». Тільки те, що відповідає Definition of Done. Зворотний зв'язок від стейкхолдерів на цій церемонії безпосередньо впливає на пріоритети Product Backlog наступного спринта.

---

## 1. Підготовка до Sprint Review (до зустрічі)

Перед зустріччю кожен сквад має підготувати **демо-пакет**:

| Що підготувати | Хто відповідає | Чому це важливо |
| :--- | :--- | :--- |
| Перелік завершених User Stories / Tasks | Scrum Master | Визначити, що саме покажемо |
| Робочий артефакт (макет, код, тест) | DEV / UX / QA | Демо має бути «живим», а не слайдами |
| Перевірка по DoD-чеклісту | QA + DEV | Незавершене — не демонструється |
| Список відкритих питань до PO | Увесь сквад | Отримати рішення, що впливає на backlog |

**Що кажемо перед зустріччю (нагадування сквадам):**
> *«Завтра Sprint Review. Демо — це НЕ презентація. Це показ працюючого результату. Якщо ваш артефакт не відповідає Definition of Done — ми його не показуємо. Краще чесно сказати 'не встигли', ніж показати напівготове і створити хибне враження прогресу.»*

---

## 2. Відкриття та контекст (5 хв)

**Що кажемо:**
> *«Ми завершили Sprint [N]. Перед нами — стейкхолдери (Product Owner / Викладач). Сьогодні кожен сквад покаже, що саме він доставив. Увага: ми демонструємо тільки те, що пройшло Definition of Done. Після кожного демо — час для зворотного зв'язку та питань.»*

**Прихована дія SM:** Нагадайте, що Sprint Review — це **діалог**, а не монолог. Стейкхолдери мають ставити питання, а команди — відповідати. Це не захист диплому, де треба «виступити красиво». Це перевірка: чи ми будуємо правильну річ?

---

## 3. Демо сквадів (30 хв — по 10 хв на сквад)

Кожен сквад демонструє свій інкремент. Порядок обирайте за пріоритетом залежностей: спочатку сквад, від якого залежать інші.

### Структура демо одного скваду (10 хв)

| Етап | Час | Що відбувається |
| :--- | :--- | :--- |
| **1. Sprint Goal** | 1 хв | SM/представник називає ціль спринта та User Stories, що були в скоупі |
| **2. Живе демо** | 5 хв | DEV/UX показує артефакт: працюючий код, інтерактивний макет, результати тестів |
| **3. DoD-чеклист** | 2 хв | QA підтверджує, які критерії DoD пройдено |
| **4. Питання стейкхолдерів** | 2 хв | PO та інші сквади ставлять питання, дають фідбек |

### Сценарії для фасилітації (Специфіка VARTA)

**Сценарій 1: Сквад Alpha демонструє EP-01 (Mesh Identity) та EP-02 (Resource Quota Ledger)**

**Що кажемо (як PO / стейкхолдер):**
> *«Покажіть мені офлайн-реєстрацію. Що бачить користувач, коли він один у Mesh-мережі — без жодного піра? Як працює Cold Start?»*
> *«Ви кажете, що є 10 тест-кейсів. Чи покривають вони сценарій, коли зловмисник намагається підписати сам себе (Self-signing)?»*

**Сценарій 2: Сквад Gamma демонструє EP-05 (Mesh Communication)**

**Що кажемо:**
> *«Я бачу макет чату з мітками 'Via 3 nodes'. Як буде виглядати стан, коли проміжний вузол вийшов офлайн? Ваш UF-02 покриває цей кейс?»*
> *«У вас 4 тікети в OPEN. Що конкретно блокує перехід від UX до Dev? Де ваші доменні правила?»*

**Сценарій 3: Сквад Beta — відсутність артефактів**

**Що кажемо:**
> *«Beta, у вас 0 тікетів у беклозі. Це факт, а не звинувачення. Sprint Review — це дзеркало: воно показує реальний стан. Розкажіть, що завадило розпочати роботу, і що ви плануєте зробити в наступному спринті, щоб це змінити.»*

**Прихована дія SM:** Не допускайте ескалації в емоційну площину. Відсутність роботи — це дані для обговорення, а не привід для покарання. Направте розмову в конструктивне русло: *«Які конкретні кроки допоможуть Beta стартувати?»*

---

## 4. Перевірка Definition of Done (5 хв)

Після демо кожного скваду проведіть **публічну звірку з DoD**. Це ключовий навчальний момент: студенти мають зрозуміти, що «зроблено» без DoD не означає нічого.

**Що кажемо:**
> *«Тепер давайте пройдемось по нашому Definition of Done. Для кожної завершеної Story — чи виконані всі критерії?»*

### DoD-чекліст для VARTA Sprint Review

| Критерій DoD | Перевірка | Хто підтверджує |
| :--- | :--- | :--- |
| Артефакт створено за шаблоном ролі (SME/UX/DEV/QA/OPS) | Є тікет у беклозі з правильним префіксом | SM |
| Acceptance Criteria виконані | AC прописані в тікеті та підтверджені | DEV + QA |
| Тест-кейси написані (для DEV-тікетів) | Існують пов'язані TC-тікети | QA |
| NFR враховані (Offline-first, RAM ≤ 1GB, Battery < 5%/год) | Зазначено в тикеті або TC | QA |
| Крос-сквад залежності задокументовані | Контракти / JSON-схеми зафіксовані в репозиторії | SM + DEV |
| Фіча перевірена в Airplane Mode | Для US, що вимагають Offline | QA |

**Якщо Story не пройшла DoD:**
> *«Ця User Story не відповідає нашому DoD — [вказуєте конкретний критерій]. Ми не приймаємо її як завершену. Вона повертається в Product Backlog і стає пріоритетом наступного спринта.»*

---

## 5. Відловлювання антипатернів Sprint Review

Зупиняйте учасників, якщо демо йде в непродуктивне русло:

| Антипатерн | Що кажете (Фасилітатор) |
| :--- | :--- |
| **«PowerPoint замість демо»**:<br>*«Ось скриншоти нашого дизайну...»* | *«Покажіть живий артефакт. Відкрийте макет у Figma, запустіть тест, покажіть код. Скриншот — це не інкремент.»* |
| **«Ми майже готові»**:<br>*«Фіча працює на 90%, залишилось трохи...»* | *«90% Done = 0% Released. Якщо вона не пройшла DoD — вона не завершена. Поверніть у backlog і завершіть наступного спринта.»* |
| **«Демо для себе»**:<br>Сквад говорить технічною мовою, незрозумілою для PO | *«Поясніть це з точки зору користувача. PO не знає, що таке 'CRDT merge'. Скажіть: 'Коли два пристрої синхронізуються — баланс оновлюється автоматично без конфліктів'.»* |
| **«Приховування проблем»**:<br>Сквад не згадує незавершені Stories | *«Я бачу в Sprint Backlog ще 3 задачі. Що з ними? Якщо вони не завершені — скажімо це чесно і обговоримо чому.»* |
| **«Відсутність питань»**:<br>Стейкхолдери мовчать після демо | *«PO, у вас є питання? Інші сквади — чи впливає те, що показала Alpha, на вашу роботу? Gamma, цей API-контракт вам підходить?»* |

---

## 6. Зворотний зв'язок та коригування Backlog (5 хв)

**Що кажемо:**
> *«Ми побачили інкремент. Тепер найважливіший момент — зворотний зв'язок від Product Owner. На основі побаченого ми можемо змінити пріоритети в Product Backlog.»*

**Питання, які PO має поставити:**
* *«Чи те, що ми побудували, відповідає тому, що потрібно користувачу VARTA?»*
* *«Чи є щось, що потрібно змінити або переосмислити?»*
* *«Чи потрібно додати нові User Stories, яких ми не передбачили?»*

**Прихована дія SM:** Фіксуйте всі рішення PO. Якщо PO каже «нам не потрібна ця фіча в такому вигляді» — це не провал команди. Це саме те, для чого існує Sprint Review: **виявити помилки дешево, а не після релізу**.

---

## 7. Закриття та підготовка до Retrospective (5 хв)

**Що кажемо:**
> *«Дякую за демо. Підіб'ємо підсумки. [Перераховуєте прийняті Stories та ті, що повернулись у Backlog]. Velocity цього спринта: [X] Story Points. Цей показник ми будемо використовувати для планування наступного спринта. Зараз ми маємо чітку картину того, ЩО ми побудували. На Retrospective ми обговоримо ЯК ми працювали.»*

**Контрольні точки після зустрічі (для викладача):**
- [ ] Чи показав кожен сквад **живий артефакт**, а не презентацію?
- [ ] Чи було перевірено відповідність DoD для кожної завершеної Story?
- [ ] Чи отримали команди конкретний зворотний зв'язок від PO?
- [ ] Чи зафіксовано зміни до Product Backlog (нові Stories, зміна пріоритетів)?
- [ ] Чи розуміють студенти різницю між Sprint Review (продукт) і Retrospective (процес)?
- [ ] Чи усвідомлюють команди, що «незавершене» — це нормально, але приховувати його — неприпустимо?

---

## Теоретичний контекст (для розбору після зустрічі)

### Sprint Review ≠ Retrospective

Найчастіша помилка починаючих команд — змішувати ці дві церемонії. Чітка різниця:

| Параметр | Sprint Review | Retrospective |
| :--- | :--- | :--- |
| **Фокус** | Продукт (що побудували) | Процес (як працювали) |
| **Аудиторія** | Стейкхолдери, PO, інші сквади | Тільки команда (внутрішня) |
| **Результат** | Оновлений Product Backlog | Action Items для покращення процесу |
| **Питання** | «Чи ми будуємо правильну річ?» | «Чи ми будуємо її правильно?» |

### Матриця демо-артефактів за ролями (довідка)

Що кожна роль може продемонструвати на Sprint Review:

| Роль | Тип артефакту | Приклад демо |
| :--- | :--- | :--- |
| **UX** | User Flow (UF), Wireframe (WF) | Інтерактивний прототип у Figma, клікабельний макет |
| **DEV** | Technical Task (TT) | Працюючий код, API-ендпоінт, unit-тести що проходять |
| **QA** | Test Case (TC) | Результати прогону тестів, покриття сценаріїв |
| **SME** | Domain Rule (DR), Edge Case (EC) | Формалізоване бізнес-правило, таблиця рішень |
| **OPS** | Ops Task (OPS) | CI/CD пайплайн, Docker-контейнер, середовище розгортання |

> 📌 **Нагадування:** Sprint Review — це не «захист проєкту». Це інструмент зворотного зв'язку. Якщо PO каже «переробіть» — це найкращий результат Review, бо ви дізнались про це зараз, а не після релізу.
