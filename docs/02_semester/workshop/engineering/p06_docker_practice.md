## Частина 1: Dockerfile для Spring Boot (20 хв)

### Бізнес-сценарій: "Works on my machine"
У вас на ноутбуці стоїть Java 21 і ви пишете код. Ви віддаєте його колезі, а у нього лише Java 17, або на сервері взагалі не встановлена Java. Результат — сервіс не стартує.
Щоб це вирішити, ми пакуємо наш застосунок разом із його оточенням (JRE) у Docker-образ. Тепер сервіс гарантовано запуститься на будь-якому комп'ютері або сервері.

### Завдання 1.1: Аналіз поганого Dockerfile (Антипатерни)
Подивіться на цей код. Це типовий Dockerfile новачка.

```dockerfile
FROM openjdk:8
COPY . /app
WORKDIR /app
RUN mvn package
EXPOSE 8080
CMD java -jar target/app.jar
```

> [!CAUTION]
> **Чому цей код ніколи не пройде Code Review:**
> 1. **Застаріла база:** `openjdk:8` — це end-of-life Java 8 без патчів безпеки.
> 2. **Відсутність кешування:** `COPY . /app` та `RUN mvn package` в одному кроці означає, що зміна будь-якого `.java` файлу скидає кеш всього шару. Це призведе до повторного завантаження десятків мегабайт залежностей Maven при кожній збірці. Щоб цього уникнути, правильно спочатку копіювати лише `pom.xml` і завантажувати залежності окремим кроком, а вже потім копіювати вихідний код.
> 3. **Гігантський розмір:** У фінальному образі лежить JDK (компілятор). Образ буде важити ~700MB, хоча для запуску достатньо JRE (~200MB).
> 4. **Немає Graceful Shutdown:** `CMD java -jar` без масиву `[]` запускається через shell. Контейнер не отримає сигнал SIGTERM при зупинці, і ваші користувачі отримають помилки під час деплою.

### Завдання 1.2: Створення оптимального Dockerfile
Створіть файл `Dockerfile` (без розширення) у корені вашого проєкту, поруч із `pom.xml`. Ми використаємо **Multi-stage build**: на першому етапі компілюємо код (потрібен важкий JDK), на другому — пакуємо лише готовий `jar` (потрібен легкий JRE).

> [!IMPORTANT]
> **Що потрібно для запуску?**
> Голої IntelliJ IDEA недостатньо! Щоб цей `Dockerfile` можна було "зібрати" та запустити, у вас на комп'ютері обов'язково має бути встановлений та працювати **Docker Desktop** (для Windows/Mac) або **Docker Engine** (для Linux). IDEA має плагіни для Docker, але під капотом вона все одно використовує встановлений на вашому ПК Docker.

**Файл: Dockerfile**
```dockerfile
# ── Stage 1: Build ─────────────────────────────────────────────
FROM eclipse-temurin:21-jdk AS builder
WORKDIR /build

# КРОК 1: Кешування залежностей. Копіюємо ТІЛЬКИ конфігурацію Maven.
# Якщо ці файли не змінилися, Docker візьме наступний крок з кешу.
COPY pom.xml .
COPY .mvn .mvn
COPY mvnw .
RUN ./mvnw dependency:go-offline -q

# КРОК 2: Збірка коду. Копіюємо вихідні файли, які постійно змінюються.
# Кеш скидається лише починаючи з цього кроку. Залежності вже завантажені!
COPY src ./src
RUN ./mvnw package -DskipTests -q

# ── Stage 2: Runtime ────────────────────────────────────────────
# Вирішення проблеми розміру: беремо легкий JRE (тільки для запуску), а не JDK
FROM eclipse-temurin:21-jre
WORKDIR /app

# Копіюємо лише jar з builder-стейджу
COPY --from=builder /build/target/*.jar app.jar

# Security hardening: Не запускаємо від root
RUN groupadd -r appgroup && useradd -r -g appgroup appuser
USER appuser

EXPOSE 8080

# Масив-форма → PID 1 отримує SIGTERM від Docker
ENTRYPOINT ["java", \
            "-XX:+UseContainerSupport", \
            "-XX:MaxRAMPercentage=75.0", \
            "-jar", "app.jar"]
```

> [!TIP]
> **Де тут магія кешування?**
> Пусті рядки у файлі — це лише візуальне форматування. Вся суть полягає у **порядку команд** (розділенні кроків).
> Кожна інструкція `COPY` або `RUN` створює невидимий "шар". Оскільки код у папці `src/` змінюється постійно, а залежності у `pom.xml` — рідко, ми спеціально **розділили їх** у Кроці 1 та Кроці 2. Коли ви зміните `.java` файл, Docker побачить, що `pom.xml` не змінився, і миттєво дістане завантажені бібліотеки з кешу, перейшовши одразу до компіляції (`COPY src ./src`), пропускаючи довге завантаження з інтернету.

> [!TIP]
> **Секрети JVM у Docker:**
> Зверніть увагу на прапорці `-XX:+UseContainerSupport` (увімкнено за замовчуванням у сучасній Java) та `-XX:MaxRAMPercentage=75.0`. Хоча сучасна Java вже "бачить" ліміти контейнера, за замовчуванням вона виділяє під Heap лише 25% від доступної пам'яті. Прапорець `MaxRAMPercentage=75.0` вказує JVM використовувати 75% контейнерної пам'яті під Heap, залишивши 25% для Metaspace та самої ОС. Без цього застосунок буде страждати від нестачі пам'яті навіть при великих лімітах у контейнері.

### Завдання 1.3: Збірка та запуск (Build & Run)

Тепер, коли у нас є ідеальний `Dockerfile`, давайте зберемо образ та запустимо контейнер. Відкрийте термінал у корені проєкту (там, де лежить файл `Dockerfile`).

1. **Збірка образу (Build):**
   ```bash
   docker build -t library-app .
   ```
   *(`-t library-app` задає ім'я вашому образу, а крапка `.` вказує на поточну папку як контекст збірки).*

2. **Запуск контейнера (Run):**
   ```bash
   docker run -p 8080:8080 -d library-app
   ```
   *(`-p 8080:8080` об'єднує порт контейнера з вашим комп'ютером, а `-d` запускає його тихо у фоновому режимі).*

3. **Перевірка:** Відкрийте браузер і перейдіть на `http://localhost:8080/api/books/ping`. Якщо ви бачите відповідь — ваш застосунок успішно працює ізольовано в контейнері!
4. **Зупинка:** Зупиніть та видаліть контейнер через графічний інтерфейс Docker Desktop або за допомогою консольних команд (див. таблицю нижче). Наступним кроком ми перейдемо до більш зручного інструменту — Docker Compose.

### Шпаргалка: Базові команди Docker CLI

| Команда | Що робить |
| :--- | :--- |
| `docker ps` | Показує список **активних** (запущених) контейнерів. |
| `docker ps -a` | Показує список **всіх** контейнерів (включаючи зупинені чи ті, що впали). |
| `docker stop <id>` | Зупиняє контейнер. Замість `<id>` можна вказати перші 3-4 символи ідентифікатора або ім'я. |
| `docker start <id>` | Запускає вже існуючий, але зупинений контейнер. |
| `docker restart <id>`| Перезапускає контейнер. |
| `docker rm <id>` | Видаляє контейнер назавжди (щоб видалити працюючий контейнер, додайте `-f`). |
| `docker logs <id>` | Показує консольний вивід (логи) контейнера. Додайте `-f` (`docker logs -f <id>`), щоб читати логи в реальному часі. |

### Завдання 1.4: Ігнорування зайвого з `.dockerignore` (Best Practice)
Коли ви запускаєте `docker build`, Docker-клієнт надсилає всі файли з поточної папки до Docker-демона (так званий *Build Context*). Якщо у вас є папка `target/` з попередніми збірками, велика папка `.git/` або конфігурації вашої IDE, передача цих даних сильно уповільнить процес.

Створіть файл `.dockerignore` у корені проєкту:

**Файл: .dockerignore**
```text
.git
.idea
target/
*.md
.env
```
> Це не тільки пришвидшує збірку, але й **запобігає випадковому потраплянню секретів** (наприклад, локального файлу `.env`) всередину публічного образу!

## Частина 2: Docker Compose (25 хв)

### Бізнес-сценарій: Infrastructure as Code
Навіть якщо наш сервіс працює in-memory і не має бази даних, запускати його через `docker run -p 8080:8080 -e APP_LIBRARY_MAX_BOOKS=50 -d library-app` дуже незручно, особливо коли параметрів стає багато. 
`docker-compose` дозволяє описати всю інфраструктуру проєкту (порти, змінні середовища, ліміти пам'яті) в одному файлі та підняти її однією командою. Це підхід **"Infrastructure as Code"**.

> [!TIP]
> **Магія Spring Boot (Relaxed Binding):**
> Звідки взялося `APP_LIBRARY_MAX_BOOKS`? У коді ми прописували `@Value("${app.library.max-books}")`. Spring Boot автоматично (неявно) перетворює змінні середовища у форматі `UPPER_SNAKE_CASE` (стандарт для Linux/Docker) на властивості `kebab-case`. Вам не потрібно писати жодного додаткового коду для цього мапінгу!

### Завдання 2.1: Створення docker-compose.yml
Напишіть `docker-compose.yml` у корені проєкту.

**Файл: docker-compose.yml**
```yaml
version: "3.9"

services:
  app:
    build: .
    ports:
      - "8080:8080"
    environment:
      APP_LIBRARY_MAX_BOOKS: ${MAX_BOOKS}
    restart: unless-stopped
    # Resource limits
    deploy:
      resources:
        limits:
          memory: 512m
          cpus: '0.5'
```

### Завдання 2.2: Змінні середовища
У `docker-compose.yml` ми використовували `${MAX_BOOKS}`. Де його взяти?

**Файл: .env (НІКОЛИ НЕ ПУШИТИ В GIT!)**
```properties
MAX_BOOKS=150
```

**Файл: .env.example (Для Git, як документація)**
```properties
MAX_BOOKS=50
```

> [!CAUTION]
> **Секрети у Git:** Якщо ви колись додасте в `.env` реальні паролі або токени доступу, і випадково закомітите цей файл на GitHub — це інцидент безпеки. Використовуйте `.gitignore`.

### Завдання 2.3: Корисні команди для відлагодження
Запустіть середовище (`docker compose up -d`) та виконайте ці команди. Вони вам знадобляться на щоденній основі:

**Оточення: Термінал**
```bash
# Статус всіх сервісів: ім'я, стан, порти
docker compose ps

# Стрімінг логів з сервісу app (Ctrl+C для виходу)
docker compose logs app --follow

# Зайти в контейнер app і переглянути його файлову систему
docker compose exec app sh

# Live-моніторинг CPU/RAM/Network по всіх контейнерах
docker stats
```

## Частина 3: Налагодження типових проблем (15 хв)

### Завдання 3.1: Діагностика інцидентів
Ось реальні ситуації з якими ви зіткнетесь у Production. Як їх вирішити?

> [!WARNING]
> **Сценарій A: App стартує і падає з помилкою `OOMKilled`**
> **Причина:** Контейнер перевищив ліміт пам'яті, вказаний у docker-compose (наприклад, `512m`).
> **Рішення:** Перевірити налаштування `MAX_RAM_PERCENTAGE` у `Dockerfile` (чи воно взагалі є) та збільшити ліміт пам'яті у Compose файлі.

> [!WARNING]
> **Сценарій B: App працює в Docker, але з хоста `:8080` дає `Connection refused`**
> **Причина:** Порт не відкритий назовні, або Spring Boot слухає лише `127.0.0.1` всередині контейнера.
> **Рішення:** Перевірити мапінг портів `ports: - "8080:8080"` у Compose. Зайти в контейнер і перевірити, чи відповідає локальний порт (`docker compose exec app telnet localhost 8080`). Якщо зсередини працює, а ззовні хоста ні — ймовірно, проблема у мапінгу, або сервер слухає лише внутрішній `127.0.0.1` замість зовнішнього `0.0.0.0`.

> [!WARNING]
> **Сценарій C: Після оновлення `.env` app все одно використовує старий ліміт книг**
> **Причина:** Docker Compose кешує контейнери, якщо конфігурація YAML не змінилась.
> **Рішення:** Перезапустити явно: `docker compose down` та `docker compose up -d` (або `docker compose up -d --force-recreate`).

## Частина 4: Оптимізація та FinOps (10 хв)

### Завдання 4.1: Порівняння розмірів образів
Розмір Docker-образу безпосередньо впливає на вартість хмарного сховища (Registry) та швидкість завантаження сервісу (Cold Start).

Якщо ви збережете "поганий" Dockerfile із Завдання 1.1 в окремий файл з назвою `Dockerfile.naive` і зберете обидва образи, ви зможете наочно порівняти їх розміри:

**Оточення: Термінал**
```bash
# (Опціонально) Збірка "поганого" образу
docker build -t library-app:naive -f Dockerfile.naive .

# Порівняння розмірів збудованих образів
docker images | grep library-app
```

> [!TIP]
> **FinOps (Економіка хмари)**
> 
> | Образ | Базовий | Розмір |
> |---|---|---|
> | Naive (JDK, без multi-stage) | `openjdk:8` | ~650 MB |
> | Optimized (JRE, multi-stage) | `eclipse-temurin:21-jre` | ~220 MB |
> | Ultra-minimal (distroless) | `gcr.io/distroless/java21` | ~130 MB |
> 
> Менший образ це не лише про гроші. Це **швидший деплой** та **менша поверхня атаки** (у distroless образах немає навіть `sh` або `bash`, тому хакер не зможе виконати шелл-команди, якщо зламає ваш додаток).

## Контрольні питання

1. **cgroups:** Ви задали `memory: 512m` у Compose, але додаток працює повільно і часто викликає Garbage Collector. Яким буде реальний розмір Heap (доступної пам'яті для об'єктів) за замовчуванням і як його оптимізувати?

<details markdown="1">
<summary>Відповідь</summary>

Сучасна JVM (починаючи з Java 8u191) коректно розпізнає ліміти cgroup завдяки увімкненому за замовчуванням `UseContainerSupport`. Вона розуміє, що її ліміт — 512 MB (а не 16 GB хоста).
Проте, через свої внутрішні налаштування (за замовчуванням параметр `MaxRAMPercentage` дорівнює `25.0`), JVM виділяє під Heap лише **25%** від цього ліміту (тобто всього ~128 MB).
Для виправлення та ефективного використання ресурсів, ми вказуємо `-XX:MaxRAMPercentage=75.0`. Тоді JVM виділить ~384 MB під Heap, а залишок залишить для Metaspace, стеків та ОС.

</details>

2. **Layer caching:** Ви змінили один рядок коду у файлі `BookService.java`. Які кроки (шари) у нашому оптимізованому `Dockerfile` будуть взяті з кешу, а які — виконані заново при `docker build`?

<details markdown="1">
<summary>Відповідь</summary>

У правильно написаному (Multi-stage) Dockerfile кеш зламається лише на етапі копіювання вихідного коду:

```text
COPY pom.xml .                 ← не змінився → БЕРЕМО З КЕШУ
RUN mvn dependency:go-offline  ← не змінився → БЕРЕМО З КЕШУ (економія часу!)
COPY src ./src                 ← ЗМІНИВСЯ    → ПЕРЕЗБІРКА
RUN mvn package                ←             → ПЕРЕЗБІРКА
COPY --from=builder ...        ←             → ПЕРЕЗБІРКА
```

Саме тому ми відокремлюємо копіювання `pom.xml` від `src`. Якби ми написали `COPY . /app` на самому початку (як у поганому Dockerfile), то кеш зламався б одразу на першому кроці, і нам довелося б заново качати весь інтернет (`mvn dependency:go-offline`).

</details>

3. **Secrets:** Ваш колега захотів передати API_TOKEN прямо у Dockerfile: `ENV API_TOKEN=secret123`. Яка проблема і як правильно?

<details markdown="1">
<summary>Відповідь</summary>

Проблема: токен назавжди вбудований у image. Видно через `docker inspect`. Залишається в history шарів навіть після `RUN unset API_TOKEN`. Image з токеном можна випадково запушити в публічний Registry.

Правильно: передавати тільки при `docker run -e` або через `.env` файл в Compose. У production — через Kubernetes Secrets, AWS Secrets Manager або хмарні Environment Variables (як у Render.com). У Dockerfile не повинно бути жодних значень — тільки `ENV API_TOKEN=""` як документація.

</details>
