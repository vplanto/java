**Тип:** Hands-on Lab
**Рівень:** Junior Strong
**Попередні вимоги:** Робочий сервіс у Git-репозиторії (GitHub/GitLab).
**Інструменти:** Render.com (Free Tier), Git.
**Інженерна мета:** Вийти за межі `localhost`. Зрозуміти концепцію PaaS (Platform as a Service) та CI/CD.

---

##  Експрес-опитування: Localhost vs Production

1.  **Availability:** Ви закрили кришку ноутбука. Що сталося з вашим сервером на `localhost:8080`? Чи доступний він клієнтам?
2.  **Access:** Ваш друг з іншого міста хоче перевірити API. Чи спрацює у нього посилання `http://192.168.0.105:8080`?
3.  **Environment:** У вас Java 21, а на сервері Java 17. Що станеться при запуску?

<details markdown="1">
<summary>Відповіді (Самоперевірка)</summary>

1.  **Він "помер".** Сервер працює, поки працює процес ОС. Немає живлення — немає сервісу. Cloud-провайдери гарантують 24/7 (SLA).
2.  **Ні.** Це ваша локальна IP-адреса всередині квартири (NAT). Щоб вас побачив світ, потрібна "Публічна IP" (Public IP) або доменне ім'я.
3.  **Crash.** `UnsupportedClassVersionError`. Це класична проблема "Works on my machine". Тому ми використовуємо Docker або чітко налаштовані PaaS-середовища.

</details>

## Частина 1: Підготовка репозиторію (10 хв)

### Бізнес-сценарій: The Cloud & PaaS
Раніше інженери купували фізичні сервери, встановлювали Linux, налаштували мережу, встановлювали Java, відкривали порти... Це **IaaS (Infrastructure as a Service)**.

Сьогодні ми використовуємо **PaaS (Platform as a Service)**.
Ви кажете хмарі: *"Ось мій код на Java"*.
Хмара відповідає: *"Я сама знайду вільний сервер, встановлю Linux, скомпілюю твій код, запущу його і дам тобі посилання HTTPS"*.

Ми використаємо **Render.com** — сучасну хмарну платформу. Але хмара не бачить файлів на вашому диску. Вона бачить ваш Git. Тому перший крок — залити код у репозиторій.

### Завдання 1.1: Git Push (Якщо ще не зробили)
Створіть репозиторій на GitHub та виконайте наступні команди у папці вашого проєкту.

**Оточення: Термінал (Локально)**
```bash
git init
git add .
git commit -m "Initial commit for deployment"
git branch -M main
git remote add origin <ВАШЕ_ПОСИЛАННЯ_НА_REPO>
git push -u origin main
```

> [!CAUTION]
> **Важливо:** Перевірте, що файл `pom.xml` лежить у корені вашого репозиторію (на першому рівні). Інакше хмара не зможе знайти інструкції для збірки.

## Частина 2: Конфігурація PaaS на Render.com (15 хв)

### Завдання 2.1: Підключення репозиторію
**Платформа: Render Dashboard**
1. Зареєструйтесь на [dashboard.render.com](https://dashboard.render.com/) (рекомендується вхід через GitHub).
2. Натисніть **New +** -> **Web Service**.
3. Виберіть **Build and deploy from a Git repository**.
4. Знайдіть свій репозиторій у списку і натисніть **Connect**.

### Завдання 2.2: Налаштування середовища (Environment)
Render запитає, як компілювати та запускати ваш додаток. Заповніть форму уважно:

* **Name:** `demo-service-<ваше прізвище>` (буде частиною URL).
* **Region:** `Frankfurt (EU Central)` — найближчий до нас регіон для найменшого Latency.
* **Branch:** `main`.
* **Runtime:** `Java`.
* **Build Command:** `mvn clean package -DskipTests`
* **Start Command:** `java -jar target/demo-service-0.0.1-SNAPSHOT.jar`
* **Instance Type:** `Free`.

> [!CAUTION]
> **Пропуск тестів:** 
> Ми використовуємо прапорець `-DskipTests` для економії часу збірки на безкоштовному тарифі. **У реальному Production так робити заборонено!**

### Завдання 2.3: Змінні середовища (Environment Variables)
У Практикумі 4 (Production Ready) ми винесли конфігурацію. Тепер ми можемо змінити поведінку програми прямо в панелі керування.

1. Прокрутіть до секції **Environment Variables**.
2. Додайте змінну:
    * **Key:** `APP_CURRENCY_USD_RATE` (Spring автоматично мапить формат `app.currency.usd-rate` у змінні середовища з великої літери та підкресленнями).
    * **Value:** `40.5` (Встановимо інший курс валют спеціально для хмари).
3. Натисніть кнопку **Create Web Service** внизу сторінки.

## Частина 3: Build, Deploy та Тестування (10 хв)

### Завдання 3.1: Моніторинг процесу збірки
Після створення сервісу автоматично розпочнеться збірка. Дивіться в консоль (Logs) прямо в браузері.

1. Render клонує ваш репозиторій.
2. Завантажує Maven-залежності (це займе 2-3 хвилини перший раз).
3. Компілює Jar файл (`Build successful`).
4. Запускає команду Java.
5. Дочекайтесь напису: `Tomcat started on port 8080`.
6. Статус сервісу зміниться на **Live** 🟢.

Угорі сторінки з'явиться ваше публічне посилання типу: `https://demo-service-xyz.onrender.com`.

### Завдання 3.2: Тестування в Інтернеті
Відкрийте термінал на вашому комп'ютері (або Postman). Ви більше не звертаєтесь до `localhost`.

**Оточення: Термінал (Локально)**
```bash
curl "https://<ВАШ_URL>.onrender.com/api/currency/convert-details?amount=100"
```

**Очікуваний результат:**
Ви отримаєте JSON з розрахунком по курсу **40.5** (який ви задали в Cloud Dashboard), а не 38.5 (який захардкоджений у вашій Java-програмі).

> [!IMPORTANT]
> **Engineering Case Study: "Cold Start"**
> Ви залишили сервіс на годину. Потім намагаєтесь відкрити посилання, і воно вантажиться 30-50 секунд замість 50 мілісекунд. Чому?
> 
> Це особливість **Free Tier** (безкоштовного тарифу). Хмара економить ресурси. Якщо до вашого сервісу немає запитів певний час, Render "присипляє" його (Scale to Zero). Перший запит "будить" процес: виділяється RAM, запускається JVM, стартує Spring Boot. Це явище називається **Cold Start**.
> У платному Production (де ви платите за 100% uptime) цього не відбувається — там завжди є "гарячий" інстанс.

## Частина 4: Завдання "На захист" (Continuous Deployment) (15 хв)

### Завдання 4.1: Тестування Auto-Deploy
1. Змініть у локальному коді `GreetingController` повідомлення на `"Hello from Cloud!"`.
2. Зробіть `git commit` та `git push` у гілку `main`.
3. Нічого не чіпайте в налаштуваннях хмари. Просто відкрийте Dashboard у браузері.

> [!NOTE]
> **Питання для захисту:**
> Що відбулося автоматично після вашого `git push` і чому це життєво необхідно для сучасного бізнесу?

<details markdown="1">
<summary>Відповідь</summary>

Render (як і будь-яка сучасна PaaS) налаштований на **Auto-Deploy** через Webhooks.
Він автоматично побачив зміни в гілці `main`, сам запустив новий білд і непомітно замінив стару версію програми на нову.
Це основа **CI/CD (Continuous Integration / Continuous Deployment)**. Головна перевага для бізнесу — радикальне зменшення "Time-to-Market" (часу від написання коду розробником до моменту, коли клієнт може цим користуватись).

</details>

##  Контрольні питання
1. **Logs:** Де тепер шукати помилки (`NullPointerException`), якщо консоль IntelliJ IDEA не показує логи сервера?
2. **Filesystem:** Ваша програма зберігає завантажені картинки у папку `images/` поруч із `jar` файлом. Після перезавантаження (Deploy) картинки зникли. Чому?
3. **Port:** У коді `application.properties` написано `server.port=8080`. Але хмара видала вам HTTPS (443). Як це працює?

<details markdown="1">
<summary>Відповіді</summary>

1. **У дашборді провайдера (Logs tab).** Або підключати централізовані системи логування (ELK, Datadog), куди сервер буде відсилати логи.
2. **Ephemeral Filesystem (Ефемерна файлова система).** У хмарі (особливо Docker/PaaS) диск очищується при кожному перезапуску. Файли треба зберігати у зовнішніх сховищах (AWS S3, Google Storage), а не на диску сервера.
3. **Reverse Proxy.** Ваш Java-додаток дійсно слухає порт 8080 всередині контейнера. Але перед ним стоїть Load Balancer хмари, який приймає HTTPS (443) і перенаправляє трафік на ваш внутрішній 8080.

</details>
