# Лекція 1a: Project Roles and Responsibilities

**Аудиторія:** 2-й курс (Junior Strong)
**Ціль:** Ознайомитися з розширеним переліком ролей в Enterprise Delivery.

> **English version:** [English](en/01a_project_roles.md) | **Незнайоме слово?** → [Глосарій курсу](glossary.md)

---

На реально великих проєктах (аутсорс або масштабні інтеграції) структура управління стає складнішою. Ось ключові ролі, з якими ви будете стикатися в Enterprise Delivery:

| Роль | Ключова відповідальність (Key responsibilities) | Метрика оцінки (Evaluation Metric) | Типова проблема (Common Pain Point) |
| :--- | :--- | :--- | :--- |
| **Account Manager** | Наглядає за клієнтом з точки зору продажів (Sales perspective): успішність поточних проєктів, пошук нових можливостей для продажу. | CSAT (Customer Satisfaction), Margin, Revenue зростання | Конфлікт між бажанням продати більше і можливостями команди це зробити |
| **Program Director** | Координатор та адміністратор усіх аспектів клієнтської програми. Керує програмою з управлінської точки зору, координує проєктні активності, ресурси та віхи (milestones) між багатьма паралельними проєктами одного замовника. | Program Delivery на час/за бюджетом | Втрата контексту деталей нижнього рівня (може приймати рішення, відірвані від технічної реальності) |
| **Engineering Sponsor** | Ескалація та координація всіх питань виключно з інженерної (Software Engineering) точки зору. | Технічна успішність, відсутність критичних архітектурних провалів | Брак часу на глибоке занурення в код, прийняття рішень на основі звітів лідів |
| **Project Manager On-Site** | Відповідає за загальне управління проєктом "на місці" (ближче до клієнта). Головна контактна особа (Principal point of contact) для клієнта. Працює з PM клієнта для узгодження планів. Уточнює функціональні вимоги, фіксує прийняті рішення та обмеження систем. | Скоуп (Scope) та строки утримано під контролем, клієнт задоволений | Клієнт постійно змінює вимоги без бажання змінювати бюджет (Scope Creep) |
| **Project Manager Off-Site** | Знаходиться разом із командою розробки в Delivery-центрі. Забезпечує технічні ресурси для проєкту, працює в парі з On-Site PM, переконуючись, що результати відповідають очікуванням. | Утилізація команди (Utilization), Delivery Predictability (передбачуваність релізів) | Розсинхронізація з On-site PM (коли команда робить не те, що зараз найважливіше клієнту) |
| **Solution Architect** | Проєктує загальну архітектуру (Solution Architecture), моніторить та керує імплементацією. Відповідає за наскрізний (End-to-End) архітектурний дизайн. | Відповідність Non-Functional Requirements (NFRs), системна стабільність | "Вежа зі слонової кістки" — малювання архітектури без перевірки можливості її реалізації розробниками |
| **Cloud Architect** | Специфічний фокус на E2E Cloud Solution. Проєктує хмарну екосистему та архітектуру в хмарі. | Cloud Bill (FinOps), Cloud Security & Availability | Over-engineering інфраструктури, перевитрата бюджету на хмару |
| **Technical Manager** | Відповідає за технічний дизайн рішення, налаштування продукту (configuration & customization). Головний технічний контакт для замовника, перевіряє що прототипи та дизайн-специфікації відповідають очікуванням. | Відповідність розробки технічному плану, відсутність технічних блокерів | Нестача часу для написання коду ("руки сверблять"), постійні мітинги замість інженерної роботи |
| **BA Lead** (Business Analyst Lead) | Відповідає за визначення вимог разом із Stream Leads та експертами. Гарантує наскрізне відстеження вимог (Requirements traceability) протягом усіх фаз проєкту. | Повнота беклогу, відсоток задач Ready for Dev | "Нескінченний" збір вимог, непорозуміння між бізнесом та інженерами |
| **Stream / Dev / Integration Lead** | Технічний лідер конкретного напрямку роботи (Stream) — наприклад, розробка, інтеграція чи міграція даних. Координує роботу розробників у своєму напрямку. | Velocity стріму, Code Quality, Delivery на час | **Постійне перемикання (Context Switching)** між кодінгом та рев'ю/менторством |
| **QA Manager / QA** | Координує роботу команди QA і несе відповідальність за якість релізу (code drop deliverables). | Escaped Defects (баги на Production), Test Coverage, Quality Gates (статус проходження перевірок) | **Постійна перевидача (перезбірка) білдів** з дрібними фіксами наприкінці спринта, тестування в поспіху |
| **SME** (Subject Matter Expert) | Предметний експерт. Надає команді глибоку експертизу в специфічній бізнес-області, що допомагає приймати правильні рішення під час розробки (Solution Delivery). | Швидкість вирішення доменних питань команди | "Вузьке місце" (Bottleneck) — усі чекають відповіді від однієї людини |

## Як вони спілкуються? (Communication Channels)

> 🔗 **Зв'язок з практикою ([Workshop: Scrum Ceremonies](workshop_02_agile.md)):**
> * **З ким спілкується Developer (Дев)?** Рядовий розробник (Backend/Mobile Dev) напряму з **PO** (або клієнтом) спілкується вкрай рідко, переважно тільки на демо/рев'ю. Щоденна робоча комунікація йде всередині команди: з **Tech Lead** або **Stream Lead** (щодо технічних рішень) та з **QA** (щодо якості та багів).
> * **З ким спілкується Project Manager (PM)?** PM зазвичай не ходить до рядового Developer'а, щоб запитати статус "нажали кнопку чи ні", або обговорити бюджет/строки. PM обговорює статус всього блоку робіт з **Technical Manager (TM)** або **Stream Lead'ом**, які вже далі управляють розробниками.
> * **Хто є "Клеєм"?** Scrum Master (або Agile Coach) — це людина, яка допомагає **всім цим ролям** ефективно спілкуватися між собою в рамках процесу, не порушуючи кордонів зон відповідальності.
