# Глосарій курсу

Якщо у лекції зустрілось незнайоме слово — шукайте його тут. Терміни згруповані за алфавітом; де є стандартний український відповідник — він вказаний у дужках.

> **English version:** [English](en/glossary.md)

---

## A

**Acceptance Criteria (AC)** (критерії приймання) — конкретний список умов, при яких User Story вважається виконаною. Формат: Given/When/Then. Договір між замовником і командою *до* початку роботи, а не перевірка після.

**Agile** (гнучка методологія) — підхід до розробки, де ПЗ будується короткими ітераціями з постійним зворотним зв'язком. Протилежність — Waterfall. Включає Scrum, Kanban та ін.

**API** (Application Programming Interface, програмний інтерфейс) — набір правил, за якими одна програма «спілкується» з іншою. Застосунок погоди не зберігає температури сам — він запитує їх через API.

**ArchUnit** — бібліотека Java для архітектурних тестів. Перевіряє: «клас у пакеті `controller` не залежить напряму від `repository`». Ломає білд, якщо хтось порушив архітектурний принцип.

**Artifact** (артефакт) — будь-який результат роботи команди: документ, JAR-файл, Docker Image, специфікація.

**Availability** (доступність) — відсоток часу, коли система доступна. 99.9% = 8.7 год. простою на рік, 99.99% = 52 хв. NFR, яку треба визначити *до* розробки.

---

## B

**BACCM** (Business Analysis Core Concept Model, модель бізнес-аналізу) — 6 взаємопов'язаних понять: Business Need, Change, Solution, Stakeholder, Value, Context. Команди, що ігнорують BACCM, часто будують «не те» — правильно, але не те.

**Backlog** (беклог) — список задач проєкту. Product Backlog — загальний. Sprint Backlog — для поточного спринту. По суті, ToDo-лист усієї команди у пріоритизованому вигляді.

**Bug** (помилка, дефект) — поведінка, яка відрізняється від очікуваної. Не плутати з фічею: іноді «помилкова» поведінка стає задокументованим функціоналом.

**Bug tracking system** (система відстеження помилок) — інструмент для реєстрації і відстеження статусу багів. Приклади: Jira, GitHub Issues, Linear, YouTrack.

**Build** (збірка) — перетворення вихідного коду на виконуваний артефакт (JAR, Docker Image). «Білд впав» = збірка не пройшла.

---

## C

**Cache** (кеш) — тимчасове сховище результатів, щоб не рахувати їх знову. Redis — популярний кеш-сервер. Аналогія: ваш мозок пам'ятає дорогу до кав'ярні і не шукає її заново щоразу.

**CAP-теорема** — фізичне обмеження розподілених систем: не можна мати одночасно Consistency, Availability і Partition Tolerance. Треба вибирати між C і A, коли мережа розривається (P неминуче).

**CAPEX** (Capital Expenditure, капітальні витрати) — великі одноразові інвестиції: купити сервер, ліцензію. Ставиться на баланс, амортизується роками. Протилежність — OPEX.

**Checkstyle** — інструмент перевірки стилю Java-коду (відступи, іменування, довжина методів). Підключається у Maven/Gradle, ломає білд при порушенні конвенцій.

**CI/CD** (Continuous Integration/Continuous Delivery, безперервна інтеграція і доставка) — **CI:** кожен коміт автоматично збирається і тестується. **CD:** після CI артефакт автоматично деплоїться. Разом перетворюють ручний деплой на конвеєр. Інструменти: GitHub Actions, Jenkins, GitLab CI.

**Clean Code** (чистий код) — код, який легко читати, розуміти і змінювати. Визначення Fowler: зміна внутрішньої структури без зміни поведінки.

**Cold Start** (холодний старт) — затримка першого запиту до сервісу, якщо він «заснув» (наприклад, у Free Tier на Render.com після 15 хвилин неактивності). Перший запит може чекати 10–30 сек, наступні — миттєво.

**Code Review** (рев'ю коду) — перевірка змін командою перед злиттям у main. Мета: знайти баги, підтримати якість і ділитися знаннями. Pull Request — механізм надсилання коду на рев'ю.

**Code Smell** («запах» коду) — ознака потенційної проблеми. Не баг, але сигнал. Приклади: Long Method, Duplicate Code, God Object, Feature Envy.

**Confluence** — вікі-система для документації команди (Atlassian). Зберігає специфікації, рішення, онбордінг. Спрощено: «спільний Google Docs для технічної документації».

**Consistency** (консистентність, узгодженість) — всі вузли розподіленої системи бачать однакові дані в один момент. CP-системи (HBase, etcd) жертвують Availability заради Consistency.

**Container** (контейнер) — запущений екземпляр Docker Image. Ізольований процес з власною файловою системою і мережею. Один сервер запускає сотні контейнерів одночасно.

**Coverage** (покриття тестами) — відсоток коду, що виконується під час тестів. 80% — типовий мінімум для нового коду. JaCoCo вимірює покриття у Java.

**CrashLoopBackOff** — стан Pod у Kubernetes: контейнер запускається і одразу падає, K8s повторює спроби з зростаючою затримкою (backoff). Діагноз: `kubectl logs pod-name --previous`.

---

## D

**Debug** (відлагодження) — пошук і усунення помилок. Дебаггер IDE дозволяє зупинити виконання в будь-якій точці і переглянути значення змінних.

**Deadlock** (взаємне блокування) — два потоки чекають один одного нескінченно. Кожен тримає ресурс, потрібний іншому. Типова проблема багатопотокового коду без правильної синхронізації.

**Demo** — демонстрація результатів Sprint замовнику або команді. По суті, показуєте «що зробили» і отримуєте зворотний зв'язок до наступної ітерації.

**Deploy** (деплой, розгортання) — запуск нової версії застосунку на сервері. «Задеплоїли на прод» = нова версія доступна реальним користувачам.

**DevOps** — культура і практики для усунення стіни між розробниками (Dev) і операційним відділом (Ops). DevOps-інженер займається автоматизацією CI/CD, інфраструктурою, моніторингом.

**Docker** — інструмент контейнеризації. Пакує застосунок разом із залежностями в Image, який однаково запускається будь-де. Вирішує «у мене працює, у тебе ні».

**Dockerfile** — текстова інструкція для збірки Docker Image: яка ОС, що встановити, що скопіювати, що запустити.

---

## E

**Environment** (середовище) — конфігурація, де запущений застосунок.
- **Dev** — локальний ноутбук розробника
- **QA / Staging** — середовище для тестування
- **Production / Prod** — реальні користувачі

**Epic** (епік) — велика User Story, що займає кілька Sprint'ів. Розбивається на менші. «Розробити модуль оплати» — епік; «Оплата карткою Visa» — User Story.

**Eventual Consistency** (кінцева консистентність) — стан AP-систем: рано чи пізно всі вузли побачать однакові дані. «Рано чи пізно» — це від мілісекунд до секунд у нормальних умовах.

---

## F

**Fail Fast** (швидко впасти) — принцип: система має виявляти помилки якомога раніше і зупинятись, а не накопичувати невалідний стан. Краще впасти при старті з чітким повідомленням, ніж мовчки видавати неправильний результат.

**Feature** (фіча, функціональність) — нова можливість у продукті. «Ця фіча з пріоритетом HIGH» = цей функціонал потрібен у першу чергу.

**Fallacy** (хибне припущення) — «Fallacies of Distributed Computing» (8 хибних припущень розподілених обчислень): «мережа надійна», «latency нульова» тощо. Peter Deutsch, Sun Microsystems.

---

## G

**Git** — система контролю версій. Відстежує зміни в коді, дозволяє повернутись до будь-якої попередньої версії. Створений Linus Torvalds у 2005 за два тижні.

**GitHub / GitLab** — хостинг для Git-репозиторіїв з Pull Request, Issues, CI/CD. GitHub — найпопулярніший публічний; GitLab — часто для enterprise з self-hosted.

---

## I

**IDE** (Integrated Development Environment, інтегроване середовище розробки) — редактор коду з дебаггером, автодоповненням, рефакторингом. IntelliJ IDEA (Java), Android Studio (Android), VS Code (universal).

---

## J

**JaCoCo** — Java Code Coverage бібліотека. Вимірює, який відсоток коду виконується під час тестів. Інтегрується з Maven/Gradle і SonarQube.

**Jenkins** — open-source CI/CD сервер. Запускає пайплайни при кожному коміті: зібрати, протестувати, задеплоїти.

**Jira** — система управління задачами та відстеження помилок (Atlassian). Тікети організовані у Board. «Закрий тікет» = позначити задачу виконаною.

**JSON** (JavaScript Object Notation) — текстовий формат обміну даними. Читабельний для людини, легко парсується. Стандарт для REST API: `{"name": "Alice", "age": 25}`.

---

## K

**Kanban** (канбан) — фреймворк гнучкої розробки без фіксованих ітерацій. Задачі переміщаються по колонках дошки (To Do → In Progress → Done). Підходить для команд підтримки де задачі «падають» хаотично.

**Kubernetes (K8s)** (оркестратор контейнерів) — автоматично запускає, зупиняє і масштабує Docker-контейнери на кластері. «K8s» = K + 8 літер + s.

---

## L

**Latency** (затримка, латентність) — час від відправки запиту до отримання відповіді. P99 latency = 99% запитів відповіли швидше за це значення.

**Legacy code** (застарілий код) — код без тестів, який важко змінювати. Не обов'язково старий: код без тестів стає legacy одразу після написання.

**Load Balancer** (балансувальник навантаження) — розподіляє вхідні запити між кількома серверами. Якщо один впав — направляє трафік на інші.

---

## M

**Maintainability** (підтримуваність) — NFR: наскільки легко вносити зміни без поломки решти. Залежить від покриття тестами, чистоти коду і документованості рішень.

**Merge** (злиття) — об'єднання змін з однієї Git-гілки в іншу. Merge conflict = Git не може автоматично вирішити, чиї зміни правильні.

**Microservice** (мікросервіс) — невеликий автономний сервіс, що відповідає за одну область. Плюси: незалежне масштабування. Мінуси: складна мережа викликів між сервісами.

**Monitoring** (моніторинг) — збір метрик (CPU, RAM, latency, error rate) і сповіщення при перевищенні порогів. Grafana + Prometheus — популярний стек.

**Mutation Testing** (мутаційне тестування) — автоматично вносить дрібні баги в код (мутації) і перевіряє, чи тести їх виловлюють. Якщо тест не впав при мутації — він неефективний.

**MVVM** (Model-View-ViewModel) — архітектурний патерн: відокремлює логіку (ViewModel) від UI (View). Стандарт для Android-розробки.

---

## N

**Namespace** (простір імен) — у Kubernetes: логічна ізоляція ресурсів у кластері. У Linux/Docker: ізолює видимість процесів, мережі, файлових систем для контейнерів.

**NFR** (Non-Functional Requirements, нефункціональні вимоги) — вимоги до *як* система працює: Scalability, Availability, Reliability, Security, Maintainability. Визначаються архітектором до розробки, не після.

---

## O

**On-call** (чергування) — розробник або SRE, що відповідає за production у позаробочий час. Якщо прод впав о 3 ночі — дзвонять on-call інженеру.

**OpenTelemetry** — vendor-neutral стандарт для збору трасування (traces), метрик і логів. Підтримує Java, Go, Python та ін. Виключає залежність від конкретного вендора (Jaeger, Zipkin, Datadog).

**OPEX** (Operational Expenditure, операційні витрати) — регулярні витрати на підтримку: хмарні сервіси, підписки, зарплати. Записується у витрати поточного року. Хмара = OPEX; власний сервер = CAPEX.

---

## P

**PACELC-теорема** — розширення CAP: враховує, що навіть *без* розриву мережі (Else) система мусить вибирати між Latency (L) і Consistency (C). Повніша модель ніж CAP для реальних систем.

**Pipeline** (пайплайн, конвеєр) — ланцюжок автоматизованих кроків: `git push` → build → test → deploy. Якщо один крок впав — пайплайн зупиняється.

**PO** (Product Owner, власник продукту) — роль у Scrum. Пріоритизує Backlog, представляє інтереси бізнесу, приймає або відхиляє результати Sprint'у.

**Pull Request (PR)** (запит на злиття) — механізм GitHub/GitLab: розробник пропонує свої зміни для об'єднання з основною гілкою. Команда робить Code Review.

---

## Q

**QA** (Quality Assurance, забезпечення якості) — ширше за тестування: процеси, що запобігають появі помилок. QA-інженер будує процеси, а не тільки знаходить баги.

---

## R

**Refactoring** (рефакторинг) — покращення внутрішньої структури коду без зміни зовнішньої поведінки. Визначення Fowler. Рефакторинг ≠ переписати з нуля і ≠ виправити баг.

**Repository** (репозиторій, репо) — директорія з кодом під управлінням Git, локальна або на GitHub/GitLab.

**REST** (Representational State Transfer) — архітектурний стиль API. Використовує HTTP-методи (GET, POST, PUT, DELETE) для роботи з ресурсами через URL.

**Retrospective** (ретроспектива) — зустріч команди наприкінці Sprint'у: що пройшло добре, що погано, що змінюємо у наступному Sprint'і.

**RPS** (Requests Per Second, запитів за секунду) — метрика навантаження. «Система витримує 10 000 RPS» = 10 тисяч одночасних запитів щосекунди.

---

## S

**Scrum** (скрам) — фреймворк гнучкої розробки. Ітерації = Sprint (1–4 тижні). Щодня — Stand-up (15 хв). PO пріоритизує Backlog. Наприкінці Sprint'у — Demo і Retrospective.

**Security** (безпека) — NFR: захист від несанкціонованого доступу, атак, витоку даних. Включає автентифікацію, авторизацію, шифрування, аудит.

**SLA** (Service Level Agreement, угода про рівень обслуговування) — договір між сервісом і клієнтом: «гарантуємо 99.9% uptime». Порушення SLA = штрафи або репутаційні втрати.

**SonarQube / SonarLint** — інструменти статичного аналізу коду. SonarQube — сервер для command, SonarLint — плагін IDE для миттєвого фідбеку. Знаходять Code Smells, Security Issues, Duplications.

**Span** — одиниця роботи у Distributed Tracing. Один HTTP-виклик, один запит до БД. Кілька Span утворюють Trace.

**Spring Initializr** — веб-генератор скаффолдингу Spring Boot проєктів (start.spring.io). Обираєте залежності — отримуєте готову структуру.

**Sprint** (спринт) — фіксований відрізок часу (1–4 тижні), де команда виконує вибрані User Stories. Закінчується Demo і Retrospective.

**Stakeholder** (стейкхолдер, заінтересована сторона) — будь-хто, хто впливає на проєкт або відчуває його наслідки: замовник, користувачі, команда, інвестори, регулятори.

**Stand-up** (стендап) — щоденна 15-хвилинна зустріч (стоячи). Три питання: що зробив вчора, що сьогодні, чи є блокери. Не місце для обговорення проблем — тільки синхронізація.

**Stateless** (без стану) — сервіс, що не зберігає стан між запитами. Кожен запит містить всю необхідну інформацію. Stateless сервіси масштабуються горизонтально без проблем.

**Strangler Fig Pattern** (патерн фікусового душителя) — стратегія заміни legacy-системи: нова реалізація будується паралельно і поступово бере на себе функціонал. Назва — від тропічної рослини, що обростає дерево і замінює його.

---

## T

**Tail Latency** — latency «хвоста» розподілу, наприклад P99 або P99.9. Якщо P99 = 200мс, то 1% запитів відповідає довше. Критично для систем де затримка одного з N мікросервісів впливає на весь ланцюг.

**TDD** (Test-Driven Development, розробка через тести) — цикл Red → Green → Refactor. Тест пишеться *до* коду. Захищає рефакторинг: якщо поведінка змінилась — тест впаде.

**Technical Debt** (технічний борг) — ціна швидких компромісів. Обрали «зробити зараз» замість «зробити правильно» — накопичуєте борг. Як фінансовий кредит: відсотки ростуть, якщо не виплачувати.

**Throughput** (пропускна здатність) — кількість успішно оброблених запитів за одиницю часу. На відміну від Latency (час одного запиту) — це загальна продуктивність системи.

**Trace** — набір пов'язаних Span у Distributed Tracing, що описує повний шлях запиту крізь систему мікросервісів.

---

## U

**Uptime** (час безперебійної роботи) — «99.9% uptime» = не більше 8.7 год. простою на рік. «Five nines» (99.999%) ≈ 5 хвилин на рік.

**User Story** (сторія) — вимога з точки зору користувача. Формат: «Як [хто], я хочу [що], щоб [навіщо]». Фокус — на цінності для користувача, не на технічній реалізації.

## V

**VCS** (Version Control System, система контролю версій) — відстежує зміни в коді, дозволяє повернутись до будь-якої попередньої версії. Git — найпоширеніший VCS.

**Velocity** (швидкість) — скільки Story Points команда закриває за Sprint. Використовується для прогнозування: «при velocity 40 ми закінчимо за 5 Sprint'ів».

**VM** (Virtual Machine, віртуальна машина) — емуляція повноцінного комп'ютера всередині фізичного сервера. Важча за контейнер, але дає повну ізоляцію ОС.

---

## W

**Waterfall** (каскадна методологія) — послідовні фази: Аналіз → Дизайн → Розробка → Тестування → Деплой. Кожна фаза починається тільки після завершення попередньої. Зворотний зв'язок — наприкінці, коли змінити щось дорого.

**WIP** (Work In Progress, робота в процесі) — кількість задач одночасно в роботі. У Kanban обмежують WIP, щоб уникнути «розпорошення» і прискорити flow.
