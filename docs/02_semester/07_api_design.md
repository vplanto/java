## 1. Відкрита дискусія (Warm-up)

**Сценарій:** Ваша команда написала API. Мобільний додаток, який його використовує, зроблений іншою командою в іншому місті. Ви вирішуєте перейменувати поле `user_name` на `username` — це ж логічніше.

Питання: що відбудеться з мобільним додатком?

<details markdown="1">
<summary>Розгорнути відповідь</summary>

Додаток впаде або почне повертати `null` замість імені користувача. Бо код клієнта очікує поле `user_name`, якого більше немає.

Якщо додаток вже опублікований в App Store — ви не можете «просто попросити користувачів оновитись». Там ще є люди на версії 6-місячної давнини.

API — це публічний контракт. Порушення контракту — це не «дрібна зміна», це зламана система для всіх клієнтів, які від вас залежать.

Сьогодні розберемо, як проєктувати API так, щоб не потрапляти в такі ситуації.

</details>

## 2. Що таке API і чому REST

API (Application Programming Interface) — контракт взаємодії між системами. REST (Representational State Transfer) — архітектурний стиль для побудови цього контракту поверх HTTP.

> 📋 **Академічна прив'язка (Системна інженерія):** З точки зору системної інженерії, API — це реалізація **Інтерфейсного проєктування (Interface Design)**. REST працює на **прикладному рівні моделі TCP/IP**, використовуючи HTTP як транспортний протокол. Ми не просто «кидаємо JSON» — ми проєктуємо **стандартизований інтерфейс обміну даними** між незалежними компонентами розподіленої системи. Цей принцип є фундаментальним для будь-якої розподіленої архітектури, яку ви будете проєктувати протягом усього курсу.

REST не є стандартом (немає RFC для REST самого по собі). Це набір принципів, описаних Роєм Філдінгом у дисертації 2000 року.

Чому REST переміг: використовує HTTP, JSON, легко кешується, людино-читабельний, не вимагає спеціальних клієнтів.

Альтернативи: GraphQL (гнучкіший, але складніший), gRPC (швидший для internal, але binary protocol), SOAP (банки та enterprise legacy — не треба плутатись).

## 3. Ресурси і методи: граматика REST

REST мислить ресурсами (іменниками), а не діями (дієсловами).

**Неправильно (RPC-стиль поверх HTTP):**
```
POST /createUser
POST /getUserById
POST /deleteUserAccount
POST /updateUserEmail
```

**Правильно (ресурс-орієнтований):**
```
POST   /users          → створити користувача
GET    /users/{id}     → отримати одного
GET    /users          → список (з пагінацією)
PUT    /users/{id}     → повне оновлення
PATCH  /users/{id}     → часткове оновлення
DELETE /users/{id}     → видалити
```

### HTTP-методи і їх семантика

| Метод | Семантика | Idempotent? | Safe? |
| :--- | :--- | :--- | :--- |
| GET | Отримати ресурс | ✅ Так | ✅ Так |
| POST | Створити ресурс | ❌ Ні | ❌ Ні |
| PUT | Замінити ресурс повністю | ✅ Так | ❌ Ні |
| PATCH | Частково оновити | ❌ Ні | ❌ Ні |
| DELETE | Видалити | ✅ Так | ❌ Ні |

**Idempotent** — повторний виклик дає той самий результат. `DELETE /users/1` викликаний двічі: перший раз видаляє, другий — 404. Кінцевий стан однаковий.

**Safe** — запит не змінює стан системи. `GET` тільки читає.

<details markdown="1">
<summary>Чому idempotency важлива в ненадійних мережах</summary>

Клієнт відправив `POST /orders` — запит пішов, але відповідь не повернулась (timeout мережі). Клієнт не знає: замовлення створено чи ні?

Якщо POST не idempotent і клієнт повторить запит — буде два замовлення. Для цього використовуються Idempotency Keys:

```http
POST /orders
Idempotency-Key: 550e8400-e29b-41d4-a716

{"product_id": 42, "quantity": 1}
```

Сервер запам'ятовує ключ. Якщо прийде повторний запит з тим самим ключем — поверне попередній результат без повторного створення.

</details>

## 4. HTTP Status Codes: відповіді зі змістом

HTTP-коди — це частина контракту. Якщо ви повертаєте 200 на помилку або 500 на помилку клієнта — ви порушуєте контракт.

### Основні групи

```
1xx — Інформаційні (Запит отримано, процес триває)
  101 Switching Protocols — зміна протоколу (напр. на WebSockets)

2xx — Успіх (Запит успішно отримано, зрозуміло та прийнято)
  200 OK             — стандартна успішна відповідь (GET/PUT)
  201 Created        — ресурс успішно створено (POST)
  204 No Content     — успішно, але тіло відповіді порожнє (DELETE)

3xx — Перенаправлення (Для завершення запиту потрібні додаткові дії)
  301 Moved Perm.    — ресурс назавжди переїхав на новий URL
  302 Found          — тимчасове перенаправлення
  304 Not Modified   — ресурс не змінювався (використовується для кешування)

4xx — Помилка клієнта (Запит містить помилки або невірні дані — ВАША сторона)
  400 Bad Request    — синтаксична помилка або невалідні дані
  401 Unauthorized   — відсутня або невірна авторизація (хто ви?)
  403 Forbidden      — авторизовано, але недостатньо прав (вам сюди не можна)
  404 Not Found      — ресурс за вказаним URL не існує
  409 Conflict       — конфлікт (наприклад, спроба створити юзера з існуючим email)
  422 Unprocessable  — дані валідні синтаксично, але не проходять бізнес-правила
  429 Too Many Req.  — перевищено ліміт запитів (Rate Limiting)

5xx — Помилка сервера (Сервер не зміг виконати коректний запит — НАША сторона)
  500 Internal Error — загальна помилка (Exception у коді сервера)
  502 Bad Gateway    — помилка проксі-сервера або шлюзу
  503 Unavailable    — сервер тимчасово не може обробити запит (overload/maintenance)
```

### Типові помилки з кодами

**Антипатерн: 200 на помилку**
```json
HTTP/1.1 200 OK
{"status": "error", "message": "User not found"}
```
Клієнт не може відрізнити успіх від помилки без парсингу тіла. Моніторинг не бачить проблем.

**Антипатерн: 500 на помилку клієнта**
```json
HTTP/1.1 500 Internal Server Error
{"message": "Invalid email format"}
```
Клієнт думає, що сервер зламався, хоча просто надіслав неправильний email.

## 5. Дизайн URL: іменники, ієрархія, консистентність

### Правила іменування ресурсів

Для того, щоб API був передбачуваним, дотримуйтесь наступних стандартів:
1. **Іменники замість дієслів:** REST оперує ресурсами. Використовуйте `/users`, а не `/get-users`.
2. **Множина (Plural):** Завжди використовуйте множину для назв колекцій. Це робить URL консистентним (`/users` для списку та `/users/123` для конкретного елемента).
3. **Kebab-case:** Якщо назва ресурсу складається з декількох слів, використовуйте дефіс для розділення (`/user-profiles`, `/order-items`). Уникайте camelCase або snake_case в URL.

```
✅  /users                 — множина для колекцій
✅  /users/{id}            — ідентифікатор у path
✅  /user-profiles         — kebab-case для довгих назв
✅  /orders?status=PENDING — фільтрація через query params
✅  /users?page=2&limit=20 — пагінація

❌  /getUser               — дієслово в URL
❌  /user                  — однина для колекції
❌  /user_profiles         — snake_case (погано читається в URL)
❌  /Users                 — великі літери
```

### Коли ієрархія — правильне рішення

Використовуйте вкладеність (Sub-resources), лише якщо дочірній ресурс існує **тільки в контексті** батьківського. 

Наприклад, замовлення належить конкретному користувачу:
```
GET /users/123/orders       — отримати всі замовлення користувача 123
GET /orders/456/items      — отримати товари в конкретному замовленні
```

**Золоте правило:** не будуйте ієрархію глибше 2-3 рівнів. 
Якщо вам потрібно щось на кшталт `/users/1/orders/2/items/3/taxes`, краще зробити `/orders/2/items/3/taxes` або навіть просто `/items/3/taxes`. Кожен ресурс має бути доступним по максимально короткому шляху, якщо він має унікальний ID.

## 6. Request та Response: що і як передавати

### JSON структура і іменування

Хоча JSON технічно дозволяє будь-які ключі, у світі REST API існують неписані стандарти:
1. **snake_case:** Найпоширеніший стандарт для JSON-ключів (`user_id`, `created_at`). Більшість публічних API (Google, Stripe, GitHub) використовують саме його.
2. **camelCase:** Часто використовується у внутрішніх JS/Java проєктах, але для публічних API вважається менш стандартним.
3. **Описовість:** Уникайте скорочень. Краще `first_name`, ніж `fn`.

```json
// ✅ Правильно: snake_case, зрозумілі назви
{
  "user_id": 42,
  "first_name": "Аліна",
  "created_at": "2024-03-15T10:30:00Z"
}

// ❌ Уникати: скорочення та змішування стилів
{
  "u_id": 42,
  "firstName": "Аліна", // camelCase краще не змішувати з snake_case
  "ts": 1710498600
}
```

### Формат помилок — консистентний

Завжди повертайте помилку в одному форматі — клієнти навчаються парсити один формат:

```json
HTTP/1.1 422 Unprocessable Entity
{
  "error": {
    "code": "VALIDATION_FAILED",
    "message": "Дані не пройшли перевірку",
    "details": [
      {
        "field": "email",
        "message": "Невалідний формат email"
      },
      {
        "field": "phone",
        "message": "Поле обов'язкове"
      }
    ]
  }
}
```

### Пагінація

Якщо ресурсів багато (тисячі користувачів), неможливо повернути їх усі одним запитом. Існує два основні підходи:

#### 1. Offset-based Pagination (Limit/Offset)
Найпростіший підхід: клієнт просить "дай мені 20 записів, починаючи з 40-го".
```http
GET /users?limit=20&offset=40
```
- **Плюси:** Легко реалізувати, можна "перескочити" одразу на 10-ту сторінку.
- **Мінуси:** На великих обсягах даних працює повільно (БД має відрахувати всі попередні записи). Якщо під час гортання сторінок додався новий запис — дані можуть "зсунутись", і користувач побачить дублікат.

#### 2. Cursor-based Pagination (Seek Method)
Клієнт просить "дай мені 20 записів після конкретного ID (курсора)".
```http
GET /users?limit=20&after_id=456
```
- **Плюси:** Дуже швидко на мільйонах записів. Стабільно: якщо дані змінюються, ви все одно отримуєте наступні записи без дублікатів.
- **Мінуси:** Не можна "перескочити" на 5-ту сторінку, можна йти тільки вперед або назад.

#### Приклад відповіді з метаданими:
```json
{
  "data": [...],
  "pagination": {
    "limit": 20,
    "has_next": true
  }
}
```

### Погляд Integration Lead: Пагінація як вузьке місце інтеграції

Коли дві системи інтегруються, саме параметри пагінації найчастіше стають точкою відмови. Проблема не в тому, що клієнт «надіслав неправильні дані» — проблема в тому, що **API-контракт не визначив поведінку для граничних значень**.

Якщо клієнт надішле `limit=-1` — що відбудеться? Якщо сервер не обробляє це явно, він може:
- Впасти з `500 Internal Server Error` (наприклад, передати `-1` напряму в SQL `LIMIT`)
- Повернути всі записи з таблиці (витік даних + OOM-помилка на великих обсягах)
- Повернути порожній результат без пояснення

Жоден з цих варіантів не є прийнятним — і **всі вони є наслідком дефекту дизайну**, а не помилки клієнта.

**Integration Lead вимагає, щоб API-контракт (OpenAPI-специфікація) явно визначав:**

| Сценарій | Обов'язкова поведінка за контрактом |
| :--- | :--- |
| `limit=0` | `400 Bad Request` з описом помилки |
| `limit=-1` | `400 Bad Request` — від'ємні значення заборонені |
| `limit=1000` (вище серверного максимуму) | `400 Bad Request` або автоматичний clamp до максимуму (залежить від домовленості) |
| Невалідний `next_cursor` | `400 Bad Request` — не `500` |

> 💡 **Зв'язок з Лекцією 8:** Конкретні тест-кейси для перевірки цих граничних значень (`limit=0`, `limit=-1`, `limit=100`, `limit=101`) — це приклад **Boundary Value Analysis (BVA)**, який ми детально розберемо в наступній лекції. Але **домовленість про те, яка поведінка є правильною** — це завдання дизайну API, а не тестування.

**Чому це важливо для інтеграції між командами (Alpha/Beta/Gamma):** Якщо Сквад Beta розробляє мобільний клієнт і надсилає `limit=500` через помилку — чи отримає він зрозуміле `400 Bad Request` від API Скваду Alpha, чи сервер «впаде» і заблокує роботу всієї команди? Відповідь на це питання має бути зафіксована в OpenAPI-специфікації **до початку розробки**.

## 7. Версіонування API: контракт не можна змінити, можна замінити

Будь-який публічний API колись потрібно змінити. Версіонування — спосіб не зламати старих клієнтів при цьому.

> 🔗 **Роль Integration Lead:** Рішення про версіонування — це не технічна деталь реалізації. Це **архітектурне рішення**, яке Integration Lead (або Stream Lead) приймає разом із командами до початку зміни. Його головний біль — запобігти ситуації, коли одна команда розгортає нову версію API, а інша команда продовжує слати запити до старої, не знаючи про зміни. Чітке версіонування та зафіксований deprecation-план — це страховка від таких «тихих поломок».

### Три підходи

**Через URL (найпоширеніший):**
```http
GET /api/v1/users
```
Плюси: очевидно, легко в документації та логах.
Мінуси: не «REST-pure» (версія не є частиною ресурсу).

**Через заголовок (Media Type Versioning):**
```http
GET /users
Accept: application/vnd.myapi.v1+json
```
Плюси: URL чистий. Мінуси: важко тестувати в браузері, менш прозоро.

**Через query parameter:**
```http
GET /users?version=1
```
Не рекомендується: query params призначені для фільтрації, а не версіонування.

### Стратегія: що є breaking change?

Breaking change — будь-яка зміна, що ламає існуючих клієнтів:

```
BREAKING (вимагає нова версія):
  - Видалення поля з відповіді
  - Перейменування поля
  - Зміна типу поля (string → int)
  - Зміна семантики існуючого endpoint
  - Видалення endpoint

NON-BREAKING (нова версія не потрібна):
  - Додавання нового поля у відповідь
  - Додавання нового endpoint
  - Додавання нового необов'язкового параметра
```

<details markdown="1">
<summary>Postels Law і чому клієнти мають ігнорувати невідомі поля</summary>

Postel's Law (принцип надійності): «Будь ліберальним до того, що приймаєш, і консервативним до того, що відправляєш».

Якщо ваш JSON-парсер падає на невідомих полях — ви порушили цей принцип. Клієнти мають ігнорувати поля, які не знають. Тоді server може додавати нові поля без breaking changes.

**У Java:** Jackson за замовчуванням може кидати помилку `UnrecognizedPropertyException`. Щоб не ламати старих клієнтів, парсер треба налаштувати на ігнорування невідомих полів через `FAIL_ON_UNKNOWN_PROPERTIES = false` або анотацію:
```java
@JsonIgnoreProperties(ignoreUnknown = true)
public class UserDto { ... }
```
Якщо ж ви явно вимагаєте нове поле через `@JsonProperty(required = true)` — ви створюєте breaking change для всіх старих клієнтів.

</details>

## 8. OpenAPI (Swagger): API як документ і контракт

OpenAPI — стандарт опису REST API у форматі YAML або JSON. З ним можна:
- Генерувати інтерактивну документацію (Swagger UI).
- Генерувати клієнтський код (openapi-generator) для Frontend/Backend.
- **Використання в QA:** автоматична генерація колекцій у Postman (через імпорт), валідація відповідей у автотестах на відповідність OpenAPI схемі.

```yaml
openapi: 3.0.3
info:
  title: User Service API
  version: "1.0"
paths:
  /users/{id}:
    get:
      summary: Отримати користувача за ID
      parameters:
        - name: id
          in: path
          required: true
          schema:
            type: integer
      responses:
        "200":
          description: OK
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/User'
        "404":
          description: Користувача не знайдено
components:
  schemas:
    User:
      type: object
      properties:
        user_id:
          type: integer
        email:
          type: string
          format: email
```

**Contract-First та роль Integration Lead:** Тут на сцену виходить Integration Lead. Його головний біль — **Context Switching та блокування команд**: Backend чекає на UX-макети, Frontend чекає на готовий API, QA чекає на обох — і спринт стоїть.

Contract-First підхід вирішує це системно: Integration Lead фіксує **YAML-файл OpenAPI-специфікації до написання будь-якого коду**. Після цього:
- **Frontend** генерує моки з OpenAPI і розробляє UI незалежно
- **Backend** реалізує ендпоінти, гарантуючи відповідність YAML
- **QA** вже на старті має готову специфікацію для написання тест-кейсів

Всі три ролі працюють **паралельно**, а не каскадом. OpenAPI-файл стає єдиним джерелом правди (Single Source of Truth) для всієї команди.

## 9. Consumer-Driven Contracts: архітектурний запобіжник від Breaking Changes

Проєктування API — це лише половина справи. Інша половина — **гарантувати**, що цей контракт не буде порушено через тиждень, коли один зі скводів вирішить «трохи підчистити» відповідь API.

### Як Integration Lead захищає контракт

Стандартне версіонування та документація OpenAPI описують контракт. Але вони не _заважають_ розробнику випадково видалити поле з JSON-відповіді. Саме тут Integration Lead вводить **Consumer-Driven Contracts (CDC)** — механізм, де відповідальність за стабільність контракту розподілена між командами.

**Ключова ідея CDC:**

> Не Provider диктує, що він повертає. **Consumer диктує, що йому потрібно.** Provider зобов'язується ці вимоги виконувати.

Якщо Сквад Beta (мобільний клієнт) потребує полів `order_id`, `status` і `total_price` — він фіксує ці очікування у **Pact-файлі**. Цей файл стає юридично-обов'язковим контрактом: API Скваду Alpha _не може_ видалити жодне з цих полів, не зламавши Pact-тест.

### Як працює Pact у CI/CD пайплайні

```
 Consumer (Frontend/Mobile)          Provider (Backend API)
 ─────────────────────────           ────────────────────────
 1. Пише Pact-тест:                  3. Запускає Pact Verifier
    "Я очікую поля                      проти свого реального API
     order_id, status,               4. Якщо поле видалено —
     total_price у відповіді"           білд Provider ПАДАЄ ❌
              │
              ▼
 2. Pact-файл (.json) публікується
    на Pact Broker (спільний сервер)
              │
              └──────────────────────►  Pact Broker повідомляє
                                        Provider про нові вимоги
```

**Результат:** Breaking Change фізично неможливо задеплоїти непоміченим. Якщо Backend-розробник зі Скваду Alpha видаляє поле `total_price` — білд Скваду Alpha червоніє ще до того, як зміна потрапить у `main`.

### CDC vs OpenAPI: різниця підходів

| Підхід | Хто гарантує стабільність | Коли ловить проблему |
| :--- | :--- | :--- |
| **OpenAPI-специфікація** | Людська домовленість + code review | Під час перевірки PR |
| **Consumer-Driven Contracts (Pact)** | Автоматизований CI/CD запобіжник | Під час білду Provider |

> 💡 **Зв'язок з Лекцією 8:** Конкретні тест-кейси для перевірки API через `MockMvc` та `@WebMvcTest` — це про те, **як** тестувати ендпоінти ізольовано. CDC — це про **архітектурну угоду між командами**, яка гарантує, що жоден тест Лекції 8 не стане зайвим через випадковий breaking change у сусідньому скваді.

## 10. Екзаменаційний пул (Exam Questions)

**Питання 1: Поясніть різницю між PUT і PATCH. Чому важливо обирати правильно?**

<details markdown="1">
<summary>Еталонна відповідь</summary>

PUT — повна заміна ресурсу. Ви передаєте всі поля. Якщо поле відсутнє в запиті — воно скидається в null або дефолт.

PATCH — часткове оновлення. Ви передаєте тільки ті поля, які змінюються. Решта залишається без змін.

Помилка: використовувати PUT для часткового оновлення. Клієнт, що хоче змінити тільки email, мав надіслати весь об'єкт — або ризикує обнулити ім'я і телефон.

</details>

**Питання 2: Поясніть різницю між 401 і 403. Наведіть конкретний сценарій для кожного.**

<details markdown="1">
<summary>Еталонна відповідь</summary>

401 Unauthorized: система не знає, хто ви. JWT-токен відсутній або протермінований. Рішення клієнта: надіслати Authorization-заголовок або перелогінитись.

403 Forbidden: система знає, хто ви, але ви не маєте прав. Авторизований менеджер регіону Lviv намагається переглянути замовлення регіону Kyiv. Рішення клієнта: немає — це не технічна помилка, а бізнес-правило. 
> 🛡️ **Security Note:** З міркувань безпеки, іноді API може повертати 404 (Not Found) замість 403 (Forbidden), щоб не видавати зловмиснику факт існування секретного ресурсу.

Плутанина між ними — поширена в проєктах без чіткої моделі авторизації.

</details>

**Питання 3: Що таке breaking change в API? Наведіть 2 приклади breaking і 2 приклади non-breaking змін.**

<details markdown="1">
<summary>Еталонна відповідь</summary>

Breaking: видалення поля `user_name` з відповіді (клієнт, що його читав, отримає null або виняток); перейменування endpoint `/users` на `/accounts` (клієнти отримають 404).

Non-breaking: додавання нового поля `middle_name` у відповідь (клієнти, що не знають про нього, просто ігнорують); додавання нового endpoint `/users/{id}/preferences` (старі клієнти не знають про нього, але не ламаються).

</details>

**Питання 4: Команда хоче змінити формат дати в API з Unix timestamp (`1710498600`) на ISO 8601 (`2024-03-15T10:30:00Z`). Це breaking change? Як провести міграцію без downtime?**

<details markdown="1">
<summary>Еталонна відповідь</summary>

Так, це breaking change: тип даних змінився з integer на string. Клієнт, що очікував число і перетворював його через `new Date(timestamp * 1000)`, отримає NaN.

Міграція без breaking change:
1. Додати нове поле `created_at_iso` поряд із `created_at` (Unix). Non-breaking — нові клієнти використовують нове поле, старі — ігнорують.
2. Задокументувати deprecation `created_at` з терміном — наприклад, 6 місяців.
3. Після 6 місяців — прибрати старе поле у v2.

Ключовий принцип: нове поле замість заміни старого, з явним deprecation period.

</details>

**Питання 5: Яку архітектурну проблему вирішує Contract-First підхід за допомогою OpenAPI в розподіленій команді під керівництвом Integration Lead?**

<details markdown="1">
<summary>Еталонна відповідь</summary>

Contract-First підхід вирішує проблему **темпоральної залежності між командами** (залежності у часі). У класичній розробці Backend-команда пише API → Frontend-команда чекає → QA чекає на обох. Весь спринт рухається каскадом, і затримка в одній ролі блокує всіх інших.

Integration Lead усуває це трьома кроками:

1. **Єдине джерело правди (Single Source of Truth):** OpenAPI YAML-файл фіксується до написання будь-якого коду. Всі три ролі (Frontend, Backend, QA) орієнтуються на один документ, а не на усні домовленості.

2. **Паралельна розробка:** Frontend генерує моки з OpenAPI і розробляє UI незалежно. Backend реалізує ендпоінти. QA пише тест-кейси. Залежності у часі зникають.

3. **Запобігання Breaking Changes до деплою:** Consumer-Driven Contracts (Pact) перетворюють YAML-специфікацію на виконуваний запобіжник у CI/CD. Якщо Provider видаляє поле, потрібне Consumer — білд падає ще в пайплайні, а не на production.

</details>
